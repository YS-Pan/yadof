# 2026-07-11 HFSS Multicore Follow-Up

## Purpose

This file consolidates the follow-up debug pass after the baseline diagnosis. It
records additional eliminated hypotheses and clarifies why owner execution is not a
deployable path.

## Direct `workflow.py` Submit Still Fails

The baseline failure was reproduced with the current submit shape:

```text
executable = workflow.py
transfer_executable = True
```

Cluster `4218` failed as `desktop-derg5ld\condor-slot1_1` in
`C:\condor\execute\dir_2368\scratch` with:

```text
analyze_setup returned False for 'Setup1'
process hf3d exited with code -1073741819
```

Conclusion: the older interpreter-as-executable submit form is not the cause.

## Owner Mode Was Blocked And Is Now Out Of Policy

A temporary owner-mode submit failed before queueing:

```text
ERROR: run_as_owner requires a valid CREDD_HOST configuration macro
```

Local configuration at the time:

- `DAEMON_LIST = MASTER COLLECTOR NEGOTIATOR SCHEDD STARTD`
- `CREDD_HOST`: not defined
- `SEC_CREDENTIAL_DIRECTORY`: not defined
- `STARTER_ALLOW_RUNAS_OWNER = False`
- `RUN_AS_OWNER = False`
- `condor_store_cred query`: `ysPan@DESKTOP-DERG5LD` has a valid password credential

The user later clarified that `run_as_owner=True` cannot be used in production
because the pool consists of many personal/office workstations with different
owners and any machine may submit work. See `../deployment_contract.md`.

Conclusion: owner mode is not a fix path for this project.

## DSO Distribution Probes

Changing PyAEDT `analyze_setup()` to pass `allowed_distribution_types=[]` did not
fix the crash. Cluster `4220` still failed with `hf3d` exit `-1073741819`.

A no-solve startup probe successfully called:

```python
set_custom_hpc_options(cores=2, tasks=1, use_auto_settings=False)
```

under `condor-slot1_1`. It set `Desktop/ActiveDSOConfigurations/HFSS` to
`pyaedt_config` and returned the generated ACF:

```text
MachineName='localhost'
NumEngines=1
NumCores=2
UseAutoSettings=False
AllowedDistributionTypes[9: 'Variations', 'Frequencies', 'Mesh Assembly','Mesher', 'Transient Excitations', 'Domain Solver', 'Solver', 'Iterative Solver', 'Direct Solver']
BoolValues(AllowOffCore=true)
```

Conclusion: the slot user can start AEDT and apply PyAEDT DSO/HPC settings. The
crash happens later in `hf3d`.

Using the slot user's real profile instead of job-local `_home/_appdata/_tmp` did
not fix the solve. Cluster `4225` still failed with `hf3d` exit `-1073741819`.

Conclusion: scratch-local profile redirection is not necessary for the crash. The
common failing condition remains slot-user execution.

## Resource Allocation Variant

A job-copy-only variant requested one Condor CPU while still asking HFSS for two
cores:

```text
request_cpus = 1
YADOF_HFSS_JOB_CPUCORE = 2
```

Cluster `4226` failed differently. Condor allocated `Cpus = 1`, and AEDT reported:

```text
Matrix solver exception: SOLVER_OUT_OF_MEMORY
process hf3d: Out of memory
```

Conclusion: Condor CPU allocation affects HFSS internals, but under-requesting CPUs
is not a valid workaround.

## `load_profile=False` Is Not Viable

With `load_profile=False`, the job failed before AEDT because Python could not
derive a username. After explicitly adding `USERNAME`, `USER`, and `LOGNAME`, PyAEDT
still warned that no installed AEDT version was discoverable from environment
variables and failed before a usable design object was available.

Conclusion: the current slot-user path requires `load_profile=True`.

## Single-Machine Observation Note

On the tested machine, collector, submit, and execute roles all ran locally. While a
job is running, live scratch can be inspected under `C:\condor\execute`. After
termination and output transfer, HTCondor may remove the scratch directory quickly.
