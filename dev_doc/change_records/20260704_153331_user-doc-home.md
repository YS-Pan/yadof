# 2026-07-04 15:33 - User Documentation Home

## Context

- The project needed a separate `user_doc/` tree for information users and AI
  assistants need while preparing optimization tasks.
- The existing `dev_doc/` tree is maintainer-oriented and should not accumulate
  detailed user workflow instructions.

## Change

- Added `user_doc/README.md` as the user-facing documentation entry point.
- Added user docs for the optimization workflow, workflow patterns, cost patterns,
  `_com.py` adapter usage, and config/run behavior.
- Updated `dev_doc/README.md` so a `dev_doc` context pass reads
  `user_doc/README.md` and follows its guide, while a `user_doc`-only pass does
  not read `dev_doc`.
- Updated architecture and the `dev_doc` blueprint to record the new documentation
  boundary.
- Added `user_doc` to terminology.

## Rationale

- User task setup needs concrete operational instructions: where to put `.aedt`
  files, how to prepare `parameters_constraints.py`, how `_com.py` files become
  active, how `workflow.py` and `calc_cost.py` are written, and how to smoke-test
  before launching optimization.
- Keeping those instructions in `user_doc/` avoids duplicating user-facing content
  across development architecture documents.

## Impact

- AI assistants reading `dev_doc/` now also read the user-facing entry point.
- Users can read `user_doc/` without pulling in framework design history.
- Future changes to task setup, adapter usage, cost calculation, config, or launch
  commands should update `user_doc/`.

## Follow-Up

- If a new simulator adapter is added under `project/com_lib/`, document its
  workflow calls in a matching `user_doc/com_lib/<adapter>.md` file.
