# 改变surrogate的运行方式

现在的surrogate有个问题是，在surrogate训练和推理期间，仿真是没有进行的，集群里的计算机闲置。

我希望修改surrogate代码，让计算机集群能被充分利用。

具体做法是，每一代完成之后，在当前机器上（submit端），先不进行代理模型训练，而是使用当前已经训练完成的“最新的”代理模型进行基于代理模型的优化（gpsaf）。生成需要用真实仿真评估的job，然后发送到集群上进行仿真。

仿真发送出去之后，在submit端开始训练新的代理模型。这样的话，正常情况下是代理模型落后真实仿真一代。如果训练耗时过长，以至于需要落后两代，这是可以接受的。如果将要落后三代，则令仿真阻塞，等训练完后再开始仿真。

现在的surrogate train 每代会被调用两次。这是为什么？是否有必要？能否缩减成只调用一次？

surrogate里应该有统计surrogate相关的metadata的代码。结果写入到recorded data里，recorded data里添加相应的接收这些信息用的api。

surrogate模块中的代码当前只集中在两个文件里，看起来有点混乱，请按功能多分几个文件。

在dev_doc\blueprints 里添加和本次修改相关的逐文件的.md文件。当前项目进行到了相对稳定的阶段，所以可以开始添加逐文件的.md文件了。如果dev_doc里有“不要写逐文件md”的指令，请去掉这些指令。
