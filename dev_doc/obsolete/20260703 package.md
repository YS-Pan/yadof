将这个项目变成一个可安装的 Python package。以后大部分稳定代码安装到 Python 环境里，用户工作目录只保存一次优化任务需要编辑的内容和运行产生的数据，不再像现在一样把框架代码、仿真文件、`workflow.py`、配置和历史数据都放在同一个源码树中。

这次改造只改变代码的部署位置、加载方式和用户入口，不改变 yadof 的核心数据链：`normalized variables -> rawData -> cost`。package 必须继续与具体任务、仿真软件和厂商无关；HFSS、其他仿真器、自定义 Python 程序以及由多个程序组成的 workflow 都应当使用同一套工作区和 CLI 机制。

本文中的两个核心名词明确限定如下：

- `package`：安装到 Python 环境中的 yadof 框架发布物。它包含跨任务复用且由项目维护者维护的核心模块、CLI、框架侧合同、默认资源和随包文档；运行时把它视为只读内容。它不代表某次优化任务，也不包含某个 workspace 的仿真文件、任务定义、历史记录或运行输出。
- `workspace`：由用户拥有、由 `yadof init` 创建或确认的一个可写目录，是一次任务配置及其运行状态的根目录。它保存用户需要提供或修改的 config、`job_template/` 与任务资源，并承载 jobs、recorded data、checkpoint、日志和工具输出。所有需要任务上下文的命令必须明确绑定一个 workspace；当前目录只是默认的 workspace 选择方式，不是 package 的一部分。

具体要求有：

1. package 的发布名、命令名和公共导入名统一使用 `yadof`。
    .1 `pyproject.toml` 需要补全 build system、项目元数据、Python 版本范围、运行依赖、可选依赖、package data 和 console-script entry point，安装后直接提供 `yadof` 命令。
    .2 最终的 package 源码建议放在 `src/yadof/`。目前 `project/` 下的 `optimize`、`evaluate_manager`、`job_template`、`recorded_data`、`surrogate`、`tools` 等模块保持原有职责和相对分组，不把它们压平到若干无关文件中；这里的“总体结构不变”指模块边界、公共 API 和数据流保持稳定，不要求继续使用含义过于宽泛的顶级导入名 `project`。
    .3 迁移过程中可以先在 `project/cli/` 实现 CLI；当核心代码进入 package 后，该模块随框架代码移动到 `src/yadof/cli/`，不会被复制到用户工作区。
    .4 框架内部导入最终统一为 `yadof.*` 或 package 内相对导入。不能依赖仓库根目录出现在 `sys.path` 中，也不能依赖 pytest 的 `pythonpath = ["."]` 才能运行。
    .5 不提供旧仓库布局或 `project.*` 公共导入的兼容层。package 化时直接把内部导入和入口切换到新结构，删除旧入口；不要为旧仓库编写迁移命令、兼容包装器或双命名空间维护逻辑。

2. 明确区分“只读的已安装框架”和“可写的用户工作区”。
    .1 安装到 Python 环境里的内容包括：优化器、任务调度和运行后端、记录与查询逻辑、代理模型、job template 的框架侧 API/数据契约/装载器、默认配置、CLI、用户工具、初始化模板和文档资源。
    .2 用户工作区包括：`config.py`、用户侧 `job_template/`、运行时生成的 `jobs/`、`recorded_data/`、代理模型 checkpoint、图表及其他任务输出。
    .3 任何运行命令都不能向 `site-packages`、package 资源目录或安装目录写入 jobs、历史记录、checkpoint、日志、图片和临时文件。package 安装目录应当可以是只读的。
    .4 所有运行路径都从明确的 workspace root 推导。默认 workspace 是当前目录，也允许所有需要工作区的命令通过统一的 `--workspace PATH` 指定；禁止用某个框架源码文件的 `Path(__file__)` 反推出用户数据目录。
    .5 一个 Python 环境应当能依次运行多个互不干扰的 yadof 工作区。加载一个工作区的 config、任务模块或历史数据后，不能通过全局模块缓存污染另一个工作区。

3. 用户工作区中的 `job_template/` 只保存任务拥有的文件。
    .1 `yadof init` 默认生成需要用户填写或经常修改的 `parameters_constraints.py`、`workflow.py`、`calc_cost.py`，以及初始化模板明确需要的任务资源。
    .2 用户可以再放入一个或多个活动的 `*_com.py` adapter、仿真器输入文件、查找表或其他 workflow 资源。`com_lib` 中的示例 adapter 可以由 CLI 按需列出和复制，但不应默认全部复制到每个工作区。
    .3 `api.py`、`parameters_constraints_class.py`、`rawdata_contract.py`、`cost_misc.py` 等框架侧文件不出现在用户工作区的 `job_template/` 中，也不要求用户维护它们；它们从已安装 package 导入，或者由 prepare-job 过程从 package 资源加入真正的 job folder。
    .4 package 内仍保留 `job_template` 模块，负责读取当前 workspace 的任务文件、提供变量/目标/归一化/成本等公共 API，并控制哪些用户文件会进入 prepared job。不能再通过导入安装目录中的某个固定 `job_template` 来代表当前任务。
    .5 prepare-job 需要把“package 提供的稳定 worker 支持文件”和“workspace 提供的任务文件”合成为自包含 job。复制规则使用明确清单或明确的排除规则，保留当前不复制 `calc_cost.py` 和不生成 `cost.json` 的边界，并支持多个 task-local adapter 和任意任务资源。
    .6 对 package 保留文件名和用户任务文件名发生冲突时，应给出可理解的错误，不能静默覆盖。prepared job 的 metadata 中记录 yadof 版本、workspace 标识、任务静态 hash 和有效配置摘要，以便日后判断结果来源。

4. 配置改为“package 默认值 + workspace 覆盖值”，而不是从已安装源码旁边寻找用户配置。
    .1 完整默认配置由 package 内部维护；`yadof init` 只在工作区生成短的 `config.py`，其中仍然只放用户常改的常量，不复制整个 `config_all.py`。
    .2 CLI 和核心模块通过统一的 workspace/config loader 取得有效配置。当前直接导入 `project.config.all`、在模块导入时固定路径的做法需要改成显式 workspace context，或者效果等价且可测试的机制。
    .3 路径类配置以 workspace root 为基准解析；绝对路径仍可由用户显式提供。错误的配置名、类型、模式和不存在的任务路径需要在开始批量计算前报告。
    .4 CLI 参数可以临时覆盖少量运行参数，例如代数、起始代数、evaluation mode 和进度输出，但不能偷偷改写 `config.py`。覆盖优先级和最终有效值要能通过命令查看。
    .5 prepared job 需要保存足够的有效配置快照和 package 版本信息以便诊断，但不要求把 package 内的整个默认配置源码复制到用户工作区。
    .6 变量、目标、rawData 语义、仿真器文件名和 adapter 选择仍属于 `job_template`，不能因为 package 化而移动进全局配置。

5. 新增统一 CLI，各种正常用户命令和现有用户工具都从 CLI 调用。
    .1 至少需要以下命令族；最终名字可以小幅调整，但职责不能丢失：

```text
yadof init [PATH]
yadof check [--workspace PATH]
yadof smoke-test [--workspace PATH] [--mode local|distributed]
yadof run [--workspace PATH] [--generations N] [--start-generation N]
yadof view cost|time [--workspace PATH]
yadof history clear [--workspace PATH]
yadof task adapters|copy-adapter|extract-parameters ...
yadof docs [dev|user]
yadof version
```

    .2 `check` 负责工作区结构、任务模块可导入性、参数/目标定义、rawData 合同、配置和所选 backend 的只读诊断；它可以报告缺失的外部环境，但不能承担管理员安装、修复 Python/HTCondor 或配置集群的职责。
    .3 `smoke-test` 运行一个明确的最小真实任务路径，并和不启动外部仿真器的 package 自测区分开。默认不应意外启动 HFSS 或其他昂贵软件；需要真实仿真时必须由用户选择相应任务或选项。
    .4 `run` 替代 `start_optimization_from_config.py` 的用途；切换时直接删除根目录 Python/`.cmd` 启动入口，业务逻辑只保留在 CLI 和 package API 中，不保留旧仓库入口的兼容包装器。
    .5 `viewCost.py`、`viewTime.py`、`clear_history.py` 和参数提取工具的用户功能迁入相应子命令。CLI 代码调用工具函数，不再通过改变当前目录、修改 `sys.path` 或启动另一个源码脚本来工作。
    .6 清理历史等破坏性命令默认要求明确确认，并提供适合自动化的显式确认参数；初始化和升级不得借此删除历史数据。
    .7 CLI 的 `--help`、错误信息、退出码和 stdout/stderr 分工保持一致。无参数或参数错误时给出可行动的中文或清晰英文提示，非交互环境中不能无限等待输入。
    .8 管理员专用的环境安装、HTCondor pool 配置和机器维护工具继续位于 `admin_tool/`，不因为“所有工具走 CLI”而混入普通用户 CLI。

6. `yadof init` 的行为需要稳定、可重复且不会破坏现有文件。
    .1 用户先创建或选择一个目录，在该目录运行 `yadof init`；也可以用 `yadof init PATH` 指定新目录。
    .2 最少生成 `config.py` 和用户侧 `job_template/`。`jobs/`、`recorded_data/`、checkpoint 等运行目录可以同时创建，也可以首次使用时延迟创建，但两种路径的最终布局必须一致。
    .3 初始化模板应当是软件无关、任务无关的最小可运行示例，不携带当前具体任务的模型文件、参数名、目标名或硬编码结果。特定仿真器示例通过显式 `--template`、`copy-adapter` 或后续复制获得。
    .4 在工作区写入一个小型机器可读标记，例如 `.yadof/workspace.json`，至少记录 workspace schema 版本和创建时的 yadof/template 版本；不要把机器专属绝对安装路径写成必须长期有效的配置。
    .5 对已经存在的目标文件，默认停止并列出冲突，不覆盖用户内容。`--force`、`--upgrade` 或其他显式模式的语义需要分别定义；即使允许更新，也应先备份或只更新能证明仍未被用户修改的模板文件。
    .6 对完整工作区重复执行 `yadof init` 应当幂等：不重置 config、不删除 task 文件、不清空运行记录，只报告已存在或补齐确实缺少的安全文件。
    .7 初始化过程尽量先在临时位置完成校验，再原子地放入目标目录，避免中途失败留下看似可用但实际不完整的工作区。

7. 需要一个明确的 workspace context 和任务装载机制。
    .1 workspace context 至少提供 workspace root、`config.py`、`job_template/`、`jobs/`、`recorded_data/`、checkpoint 和工具输出路径，供各核心模块通过公共接口使用。
    .2 `parameters_constraints.py`、`calc_cost.py` 和其他 submit-side 任务模块从指定工作区加载；`workflow.py` 和 task-local adapter 作为 job payload 使用。加载过程需要支持这些任务文件之间合理的本地导入。
    .3 不要把工作区永久插入全局 `sys.path`。如果使用 `importlib` 动态加载，需要为每个工作区建立隔离的模块命名空间，并定义 reload/cache 失效规则，使用户在两代之间修改允许修改的任务文件后能够得到当前版本。
    .4 `optimize`、`evaluate_manager`、`recorded_data` 和 `surrogate` 仍通过公共 API 协作，但公共 API 需要能够使用当前 context，而不是隐式绑定安装包内的固定 task/config/data 路径。
    .5 中途修改参数范围、workflow、仿真输入或 `calc_cost.py` 时，继续遵守现有动态归一化、动态 cost 和 `job_static_hash` 规则；package 化不能把 cost 变成持久化真值。

8. 所有持久化数据都相对于工作区保存，并保持现有合同。
    .1 jobs、`indMeta.jsonl`、`rawData.npz`、`optMeta/`、代理模型 checkpoint 和可视化输出的含义不变，只把根路径从源码 package 移到 workspace。
    .2 本地和 distributed backend 使用相同的 workspace/job/recording 合同。HTCondor worker 侧需要能够运行 package 提供的 bootstrap 和 workspace 提供的任务 payload，但不能要求共享可写的 `site-packages`。
    .3 distributed job 必须记录并回传足够的 package/workspace/task 版本信息；worker 上 package 缺失或版本不兼容时应当形成可诊断的 job failure，而不是在结果收集阶段表现为不明文件缺失。
    .4 历史数据的锁、原子写入和失败隔离规则保持不变。多个工作区的同名 job 不得写入同一记录目录。

9. `dev_doc/` 和 `user_doc/` 必须随 package 保留并能被可靠访问。
    .1 源码仓库仍只维护一份权威文档，构建时把两棵文档作为 package data 包含进 wheel 和 sdist，不能依赖安装完成后仍存在 Git 仓库相对路径，也不要人工同步两份容易漂移的副本。
    .2 package 代码通过 `importlib.resources` 或等价的资源 API 查找文档和初始化模板，不能假设资源一定是普通可写文件。
    .3 `yadof docs user` 和 `yadof docs dev` 至少能打印文档入口的实际位置或内容；是否调用图形程序打开文档必须是显式选项，默认命令保持可在无 GUI 环境使用。
    .4 package 化完成时同步更新 `user_doc` 中的安装、初始化、配置、smoke test、启动和工具用法，并更新 `dev_doc` 中受到影响的架构、蓝图、术语和变更记录。

10. 依赖需要按功能分层，避免把某个仿真器变成 yadof 的强制依赖。
    .1 核心运行依赖写入 package 的基础 dependencies；代理模型、绘图、开发测试和特定 simulator adapter 的额外依赖按实际边界放入 optional dependency groups。
    .2 HTCondor 可执行程序、仿真器本体和系统环境仍由管理员准备，不能假装成普通 pip 依赖。CLI 只检查和报告它们是否可用。
    .3 安装基础 package 后应能运行 `yadof --help`、`yadof init`、结构检查和不依赖外部仿真器的本地最小流程。
    .4 package 元数据中声明版本，并提供单一的运行时版本来源；job metadata、workspace marker、`yadof version` 和构建产物使用同一个版本值。

11. 实现顺序应当让 package 和工作区边界尽早可测试。
    .1 先建立 package 构建、版本、console entry point、资源读取和最小 CLI。
    .2 再实现 workspace context、config loader、task loader 和 `yadof init`，用一个通用的纯 Python 模板跑通本地最小闭环。
    .3 然后逐个迁移 `recorded_data`、`evaluate_manager`、`optimize`、`surrogate` 和工具的路径/导入依赖，保持每一步仍有合同测试。
    .4 再迁移 distributed backend 和真实 task adapter；最后删除根启动脚本业务逻辑、旧 `project.*` 导入和源码目录假设。旧仓库数据及布局不在实施范围内。
    .5 每个阶段都按 `dev_doc/README.md` 更新当前架构、对应蓝图、用户文档和 change record；不要等所有代码移动完再集中修文档。

12. 测试不能只在源码仓库根目录里通过。
    .1 构建 wheel 和 sdist，检查其中包含 CLI、核心模块、初始化模板、`dev_doc`、`user_doc` 和必要 package data，不包含当前具体任务模型、运行历史、jobs、checkpoint、缓存或密钥。
    .2 在干净虚拟环境中安装 wheel，从仓库外的临时目录运行 `yadof --help`、`yadof version`、`yadof init`、`yadof check` 和纯 Python local smoke test。
    .3 验证初始化结果不含 `api.py` 等框架文件，重复 init 不覆盖用户修改，冲突/升级行为符合定义，package 目录只读时所有正常命令仍可运行。
    .4 验证两个 workspace 在同一个 Python 进程或连续 CLI 调用中使用各自的 config、task、records 和 checkpoints，没有 import/cache/path 串扰。
    .5 验证 prepared job 同时包含必需的 package worker 支持和 workspace task payload，仍然只产出 rawData/metadata，不保存权威 cost。
    .6 验证 local 与 mocked distributed 流程、失败/timeout、动态 cost/归一化、同一 workspace 内的历史恢复、代理模型 checkpoint 和 CLI 工具。
    .7 至少有一个与 HFSS/Ansys 无关的通用任务测试，证明基础 package 没有绑定单一 simulator、厂商、模型文件或 adapter。
    .8 安装态测试不能通过额外设置仓库 `PYTHONPATH`、从源码目录导入未打包文件或依赖当前工作目录碰巧是仓库根目录来通过。

13. 完成的判断标准是：
    .1 用户在一个干净 Python 环境安装 yadof 后，可以在任意可写目录执行 `yadof init`，只编辑该工作区的 `config.py` 和 `job_template/`，然后通过 CLI 检查、smoke test、启动/恢复优化并查看结果。
    .2 框架代码和文档来自已安装 package；用户工作区不含 `api.py`、优化器、调度器、记录实现、代理模型实现等稳定源码，也不向 package 目录写数据。
    .3 当前 local、distributed、记录、动态 cost、代理模型和失败隔离合同在新布局下继续成立；不要求旧仓库或旧工作目录能被新 package 读取或迁移。
    .4 wheel/sdist 内容测试、干净环境安装测试、仓库外 CLI 集成测试和默认 pytest 均通过。
    .5 所有用户入口文档已经改成 package + workspace + CLI 用法，相关架构/蓝图/术语已经更新；完成这些代码和文档工作后，再把本文件移动到 `dev_doc/obsolete/`。
