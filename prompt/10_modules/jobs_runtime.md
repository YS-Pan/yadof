# Module prompt: jobs runtime

## Intent
- Represent per-individual execution sandboxes created by `evaluate_manager`.
- Keep job folders disposable after their rawData and metadata have been recorded.
- Support both local execution today and future distributed execution with the same output contract.

## Functionalities
- Hold copied workflow/runtime files needed to run one evaluation.
- Hold `job_input.json` with normalized and unnormalized variables.
- Hold `metadata.json` and `metaData.json` with lifecycle status and diagnostic information.
- Hold flat `rawData/*.npz` files generated by the workflow.
- In distributed mode later, provide the filesystem rendezvous point where cluster jobs write outputs.

## I/O Format
- Input files are copied from `job_template` plus generated job input/metadata.
- Output files are metadata and `.npz` rawData files only.
- Cost files such as `cost.json` are outside the v3 contract.

## Non-Obvious Techniques
- Jobs are not a Python package or active module; they are runtime state.
- The static hash is calculated before per-individual runtime outputs exist.
- Recorded-data ingestion should be able to consume job outputs even when the job status is error or timeout.

## Mutability Profile
- Job folder contents are generated artifacts and may be deleted or moved after recording if a cleanup policy is added.
- The layout should stay stable enough for local and distributed backends to share finalization logic.
